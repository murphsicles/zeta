## Zeta Bootstrap Deep Dive - Critical Architecture Discovery
**Date:** 2026-03-19 21:56 GMT
**Session:** Pre-compaction memory flush

### 🎯 **CRITICAL BREAKTHROUGH: Understanding the Real Bootstrap Challenge**

After 5+ hours of intensive debugging and analysis, we've uncovered the **fundamental architectural issue** preventing Zeta v0.5.0 bootstrap:

### **Key Discoveries:**

1. **v0.5.0 "Zeta Source" is Rust Code in Zeta Clothing**
   - The `src/` directory contains 49 `.z` files that are **Rust compiler source translated to Zeta syntax**
   - Uses Rust patterns: `Result<T, E>`, `unwrap()`, `?` operator, `if let`, `HashMap`, `String`, etc.
   - This is a **bootstrap test**, not native Zeta compiler code

2. **v0.3.6 Compiler Has Fundamental Expression Handling Bugs**
   - **`if` expressions don't work**: `if x == 1 { x } else { 0 }` compiles but returns 0 instead of correct value
   - **Root cause**: MIR generation doesn't capture expression values from blocks
   - **Architectural flaw**: `MirStmt::If` has no `dest` field to store result; blocks as expressions not properly supported

3. **Parser Actually Works (Partially)**
   - v0.5.0 source partially parses (15 ASTs parsed successfully)
   - Type checking passes
   - **MIR generation fails silently**: "mir_map len: 0" - empty MIR map despite successful parsing

4. **The Real Issue Chain:**
   ```
   Expression-oriented Zeta code
   → Parser accepts it  
   → MIR generation fails to capture expression values
   → Codegen has nothing to generate
   → Linking fails (no entry point)
   ```

### **Technical Details Uncovered:**

1. **MIR Generation Bug in `src/middle/mir/gen.rs`:**
   - `AstNode::If` handling creates new `MirGen` instances for `then`/`else` blocks
   - These sub-generators don't return values to parent
   - `MirStmt::If` has no `dest` field to store expression result

2. **Codegen Bug in `src/backend/codegen/codegen.rs`:**
   - `MirStmt::If` implementation creates basic blocks but **no phi node** for merged values
   - Control flow merges without capturing which value to use

3. **Language Design Reality:**
   - Zeta v0.3.6 appears to have **statement-only `if`** based on AST definition
   - But parser accepts expression `if`, creating inconsistency
   - Blocks (`AstNode::Block`) exist but value semantics unclear

### **Strategic Realization:**

The bootstrap challenge is **NOT** just fixing compilation errors. It's about:

1. **Making v0.3.6 support expression-oriented Zeta** (blocks as expressions, `if` returning values)
2. **OR** Creating a minimal Zeta compiler in simple Zeta (bypassing Rust-like code)
3. **OR** Fixing the architectural mismatch between parser and MIR/codegen

### **Critical Decision Point:**

User directive: **"No half measures. No technical debt."** We must:
- Fix the root cause completely
- Not patch or work around
- Implement proper expression support in MIR and codegen
