

## DEEP DIVE: v0.3.7 Architecture Understanding (2026-03-20 22:25-22:39 GMT)

### 🕵️‍♂️ Critical Discovery: Function Call Transformation Pipeline

Through deep debugging with added instrumentation, we discovered the exact transformation:

**`add_one(41)` → AST: `add_one.call(41)` → MIR: `call_i64(add_one, 41)`**

**Debug output:**
```
[MIR GEN DEBUG] Processing call: method="call", receiver=Some(Var("add_one")), args=[Lit(41)]
[MIR GEN DEBUG] Method call, mangled to: call_i64
[MIR DEBUG] Stmt[0]: Call { func: "call_i64", args: [2, 3], dest: 1, type_args: ["i64"] }
```

**Architecture Insight:** In v0.3.7 Zeta:
- All function calls are **method calls** on function objects
- `add_one(41)` is parsed as `add_one.call(41)` (function object with `call` method)
- The `call` method gets mangled with type (`i64`) to `call_i64`
- This enables **first-class functions** - functions are objects with `call` method

### 🛠️ Attempted Fix: Direct Call Generation

We attempted to modify MIR generation to detect `method == "call"` with `receiver` being a function name (`Var("add_one")`) and generate direct calls instead of `call_i64`.

**Implementation attempted:** In `src/middle/mir/gen.rs`, added special handling in `AstNode::Call` match arm to check for `method == "call"` and `receiver` being `AstNode::Var`. If true, generate `MirStmt::Call` with `func: func_name` directly instead of mangling to `call_i64`.

**Technical Challenge:** Rust pattern matching issues with `Option<Box<AstNode>>` and `&Box<AstNode>` dereferencing. Multiple attempts failed due to "cannot explicitly borrow within an implicitly-borrowing pattern" errors.

### 🔄 Alternative Approaches Considered

2. **Fix codegen to handle `call_i64` properly** - Implement proper function pointer casting in LLVM IR generation
3. **Add `call_i64` to runtime** - Implement as built-in primitive that dispatches calls via function table

**Current Status:** At 22:39 GMT, we're at decision point. The architecture is now understood, but implementation requires careful Rust pattern matching or alternative approach.

### 📊 Key Learnings

1. **v0.3.7 is a functional language** - Functions are first-class objects with `call` method
2. **Mangling transforms `call` to `call_i64`** - Type specialization happens at MIR generation
3. **Bootstrap must work within constraints** - Either fix MIR generation or implement runtime support
4. **Pattern matching in Rust is tricky** with nested `Option<Box<T>>` types
